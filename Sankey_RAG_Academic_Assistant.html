        return

    files = [f for f in os.listdir(directory) if f.endswith('.pdf')]

    print(f"📊 Analyzing {len(files)} documents for structural integrity...")
    
    for file in files:
        path = os.path.join(directory, file)
        try:
            with pdfplumber.open(path) as pdf:
                sample_page = pdf.pages[0]
                tables = sample_page.extract_tables()
                text = sample_page.extract_text()
                
                audit_results.append({
                    "Document": file,
                    "Pages": len(pdf.pages),

                    "Is_Searchable": text is not None and len(text.strip()) > 50

🧩 Document Chunking (Redefining Chunks)¶
This phase transforms the raw, extracted PDF text into manageable semantic chunks for indexing inside the vector database.


Selecting the right chunking strategy is critical for maintaining the technical integrity of complex Operating Systems concepts such as:


    

  • Deadlocks
    • 

  • Virtual Memory
    • 

  • Process Scheduling
    • 

  • Paging & Thrashing
    • 



Poor chunking can break logical continuity, leading to incomplete or inaccurate retrieval results.




🔎 Technical Approach¶
🔹 1. Recursive Splitting¶
The system uses RecursiveCharacterTextSplitter to split documents intelligently.


Instead of cutting text at arbitrary fixed lengths, this method:


    

  • Prioritizes paragraph boundaries
    • 

  • Falls back to sentence boundaries if needed
    • 

  • Maintains semantic grouping of related concepts
    • 



This ensures that tightly coupled explanations (e.g., the four Coffman conditions for deadlock) remain intact within the same chunk whenever possible.




🔹 2. Context Preservation Strategy¶
To balance retrieval precision and contextual depth:


    

  • Chunk Size: 1000 characters
    • 

  • Chunk Overlap: 100 characters
    • 



The 100-character overlap ensures:


    

  • Definitions are not split mid-explanation
    • 

  • Multi-step technical lists remain connected
    • 

  • Edge-of-chunk information is preserved
    • 



This significantly improves downstream answer synthesis quality.

This phase transforms processed text chunks into high-dimensional vector representations and stores them in a local database.

Because academic PDFs may contain malformed encodings or corrupted characters, a resilient batching strategy is implemented to prevent full pipeline failure.

🔹 Local Embeddings¶
    

  • Uses the all-MiniLM-L6-v2 model from Hugging Face
    • 

  • Lightweight and optimized for local deployment
    • 

  • Converts each text chunk into a 384-dimensional vector
    • 

  • Efficient for semantic similarity search in academic contexts
    • 



This enables fast embedding generation without requiring large GPU resources.




🔹 Resilient Vectorization Strategy¶
Instead of bulk insertion, the system uses an individual-insertion loop:


    

  • Processes chunks one at a time
    • 

  • Skips problematic chunks safely
    • 

  • Prevents a single encoding error from crashing the entire build process

This "error-skipping" logic ensures robustness during database creation.




🔹 Persistent Local Storage¶
    

  • Uses ChromaDB for vector storage

This significantly improves system startup efficiency.

# 2. Re-validate data types strictly
texts = [str(doc.page_content) for doc in chunks]
metadatas = [doc.metadata for doc in chunks]

# 3. Resilient Vectorization
print(f"⚡ Vectorizing {len(texts)} chunks with error-skipping...")

vectorstore = Chroma(
    collection_name="os_rag_final_resilient",
    embedding_function=embeddings_model,
    persist_directory="./chroma_db_local"
)

success_count = 0
fail_count = 0

# We process individually for maximum safety since batches are failing
for i in range(len(texts)):
    try:
        # Check if text is actually a string and not empty
        current_text = texts[i]
        if isinstance(current_text, str) and len(current_text.strip()) > 5:
            vectorstore.add_texts(texts=[current_text], metadatas=[metadatas[i]])
            success_count += 1
    except Exception:
        fail_count += 1
        # Silent skip to keep the console clean, or print(f"Skipping chunk {i}")

    if (i + 1) % 250 == 0 or (i + 1) == len(texts):
        print(f"📦 Progress: {i + 1}/{len(texts)} (Success: {success_count} | Skipped: {fail_count})")

# 4. Setup Retriever
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})
print(f"\n✨ FINISHED! Vector store ready with {success_count} valid OS segments.")

def build_vector_store_manually(splitter, docs_to_process):
    chunks = splitter.split_documents(docs_to_process)

    final_texts = []

        texts=final_texts,

        metadatas=final_metadatas

    return db

The four necessary conditions for a deadlock in a system using either deadlock prevention or deadlock avoidance are:
1. Resource R1 is not yet available to any process, and there is no waiting process.
2. Resource R3 is already allocated to some other process.
3. Process P1 holds the lock on resource R1 (ie, can see it but cannot modify/delete it), and 
process P2 holds the lock on resource R2 (ie, can see it but cannot modify/delete it).
4. Resource R2 is not yet available to any process.

The four necessary conditions for deadlock prevention are:
1. All requests for resources must be fulfilled before any process begins execution.
2. Processes P1 and P3 cannot simultaneously hold any resource (ie, no lock is required).
3. Process P4 may relieve the hold on R2 if it holds a different resource that is not yet available to 
any process.

Without these conditions, the system can still have deadlocks. However, with deadlock prevention in place, 
the system guarantees that at least one of these four necessary conditions cannot hold. This ensures that 
deadlocks are impossible and allows the system to function correctly.

Virtual memory utilizes demand pagging to ensure that a page is not loaded from secondary storage until it is needed for execution. When a program is running, its code and data are stored in the main memory (RAM) where they are accessed by the CPU. Demand pagging involves swapping out pages from the secondary storage (such as a flash drive or hard disk) when they are needed for execution. The second storage is used for storing temporary data until the page is actually requested, which can help reduce the number of virtual page faults that occur during the actual process execution.

Overall, demand pagging is an important aspect of virtual memory management, as it ensures efficient use of available resources and prevents fragmentation from occurring.

--- 🚀 EXPERIMENT (Recursive) ---

When a page of data is needed, the operating system loads it into the frame with the smallest number of pages. Once all of those pages are used, the remaining free pages are loaded in the next frame until the entire logical address space has been utilized. This process of loading and unloading memory pages optimizes the allocation of physical memory, reducing external fragmentation and improving overall system performance.

TESTING QUESTION: What is the difference between a mutex and a semaphore?
🔍 Processing: What is the difference between a mutex and a semaphore?

--- 🧱 BASELINE (Fixed) ---
A mutex (from Greek "mutex" which means "latch") is a locking mechanism that prevents multiple processes from accessing a shared resource simultaneously. It works by assigning a unique identifier to each process, and then using a semaphore to control access to the resource. A semaphore has the ability to count up to a specific value (usually 1 or 0), which allows it to be used as a mutex locking mechanism.

In contrast, a semaphore is not necessarily exclusive - in fact, semaphores can have arbitrarily high values that allow multiple processes to simultaneously use the resource. In this case, the resources can be viewed as being available for use by each process independently of the others. This is why semaphores are sometimes also known as "pools" or "locking mechanisms."

In contrast, a mutex is exclusive - in this case, only one process can simultaneously access the resource at any given time. Mutexes allow for higher levels of concurrency and parallelism, making them particularly useful in systems with many resources that need to be accessed by multiple processes at once.

--- 🚀 EXPERIMENT (Recursive) ---
A mutex and a semaphore are different types of locking mechanisms used to ensure exclusive access or control over a resource.

Mutex: A mutex is a lock that is held by the calling thread until it has completed its operation or another thread has obtained a specific value for the mutex (e.g., 0, 1, etc.). The value of the mutex indicates whether the resource has been successfully locked for the calling thread.

Semaphore: A semaphore is a counting variable that increases by 1 as soon as another thread acquires the mutex and decreases by 1 when another thread releases it (or the lock is broken, if any). This means that the maximum number of concurrently accessing this resource is determined by the value of the semaphore.

In a writer process (i.e., a process writing to a shared resource), the mutex and the semaphore are typically used in the same way. The mutex ensures exclusive access to the resource, while the semaphore indicates how many processes are currently accessing it. When the first process enters the critical section (the writer's segment of code), it waits on the semaphore until all other threads have released the lock before it can proceed with its work.

When a reader process is entered into the critical section, it blocks waiting on the mutex to ensure that only writers are accessing the resource at that time. The reader's segment of code then releases the lock and waits for any writer who has acquired it before it can proceed with its own work. If there are no writers currently blocking on the same mutex, the process will sleep until another writer requests access to the resource.

In summary, a mutex is used to ensure exclusive access to a shared resource, while a semaphore is used to count the number of concurrently accessing that resource.

TESTING QUESTION: Describe the concept of 'Context Switching'.
🔍 Processing: Describe the concept of 'Context Switching'.

--- 🧱 BASELINE (Fixed) ---
Context switching is the transfer of control from one process to another for the purpose of saving or restoring context. It involves saving the current process's state, including memory management information and CPU scheduling information, and loading a different process's state that is ready to run on the next cycle. There are several methods to switch context, such as software-based switching, hardware-based switching, and hybrid switching. The timing and duration of context switching can vary depending on the machine speed and number of registers used. In multiprogramming systems, the objective is to optimize CPU utilization by switching among multiple processes at different times.

--- 🚀 EXPERIMENT (Recursive) ---
Context switching refers to the process by which a program or process switches its control from one process to another during execution, to accommodate for other processes' demands. The process is known as "context switching" because it involves moving the CPU between two different contexts or states (i.e., the current running process and the next ready-to-run process). In an operating system, this switch happens every time a process finishes its execution, either due to a signal from another process, or because of resource exhaustion or other reasons.

When a context switch occurs, the CPU is saved in the PCB (Program Counter Bank) and transferred to another process's PCB. The current running process's information, including its memory allocation, registers, and other state information is saved on the CPU's stack for use during the next execution cycle.

During a context switch, the kernel saves the previous process's PCB information (including the current time, user input/output, system calls made by the previous process, etc.) in the memory-mapped "stack pointer" (the base address of the stack) before switching to the next process. Once the new process is loaded into memory, it's restored from the saved state using a similar technique.

When multiple processes are running at once, there are typically multiple context switches occurring simultaneously, each moving between the current and next available process's PCB information. The switching periods depend on several factors, including the speed of the CPU (the higher the speed, the fewer context switches occur), the number of registers used in each process (the more registers used, the longer the transition), and the amount of memory allocated to each process (the more memory allocated, the longer the transition).

In a uni-processor system, only one process can run at any given time, and switching between processes requires no more than one context switch. When operating on multiprocessor systems, multiple processes may run simultaneously and switch among them based on resources available. In this case, each running process must wait until it's notified by the other process that it's ready to run, or until a high priority process enters into memory and switches to the next process.

Overall, context switching is necessary for efficient use of system resources, preventing potential issues such as system malfunction, resource exhaustion, and program crashes.

A page fault occurs when a program accesses a non-existent or invalid memory location during execution. It is caused by a virtual address (VA) collision between different pages in the memory hierarchy, leading to a segmentation fault (SegFault). The Page Fault handler, which handles page faults and marks invalid regions as unmappable, is responsible for bringing those regions into physical memory, before terminating the process or calling other interrupt services. In a more detailed manner, when a process refers to a non-existent or invalid memory location, it checks an internal table (page table) stored in memory. If the reference is valid but not yet brought in, the page fault handler triggers a disk read operation to read the desired page from disc and bring it into physical memory. Once in memory, the page is modified in the Page Fault handler to indicate that it is now available for use. This process of bringing the non-existent or invalid region into physical memory is handled by the Page Fault handler, which also marks invalid regions as unmappable.

A page fault occurs when a process tries to access a non-existent or inaccessible memory location, which causes a hardware interrupt (or other signal) to be generated.
